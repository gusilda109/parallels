# Лабораторная работа №2: OpenMP — метод простой итерации

Параллельная реализация решения СЛАУ `Ax = b` методом простой итерации
с использованием OpenMP. Два варианта параллелизации:
- **V1**: на каждый распараллеливаемый цикл — своя `#pragma omp parallel for`.
- **V2**: одна `#pragma omp parallel` на весь итерационный алгоритм.

## Структура

```
lab2/
├── Makefile
├── src/
│   ├── serial.cpp        # последовательная версия (T(1))
│   ├── parallel_v1.cpp   # V1
│   └── parallel_v2.cpp   # V2
├── scripts/
│   ├── run_scaling.sh    # замеры масштабируемости (V1 vs V2)
│   ├── run_schedule.sh   # исследование schedule(...)
│   └── plot.py           # построение графиков
└── results/              # сюда пишутся CSV и PNG
```

## Исходные данные

Модельная задача с заданным решением:
- `a_ii = 2.0`, `a_ij = 1.0` (i ≠ j)
- `b_i = N + 1`
- начальное приближение `x_i = 0`
- решение: `x_i = 1` для всех i

Параметры метода: `τ = 0.0001`, `ε = 1e-5`. Матрица плотная (хранится целиком).

## Запуск на удалённом сервере

### 0. Загрузка и сборка

```bash
# Скопировать архив на сервер, распаковать
scp lab2.tar.gz user@server:~
ssh user@server
tar xzf lab2.tar.gz && cd lab2

# Собрать
make
```

> Требуется `g++` с поддержкой OpenMP (есть везде) и Python 3 + matplotlib
> для графиков (`pip install --user matplotlib`).

### 1. Подбор размера задачи (≥ 30 с на 1 ядре)

```bash
./bin/serial 15000     # для машины с быстрым CPU начните с 15000
# Если меньше 30 с — увеличьте до 18000 или 20000.
# Если 20000² × 8 B = 3.2 ГБ — проверьте, что памяти хватает.
```

### 2. Замеры масштабируемости (пункт 2 ЛР)

```bash
./scripts/run_scaling.sh 15000 3
# 15000 — N, 3 — число повторов (берётся минимум по времени)
```

Замеры идут на 1, 2, 4, 7, 8, 16, 20, 40 потоках для V1 и V2.
Результаты — в `results/scaling.csv`.

### 3. Графики

```bash
python3 scripts/plot.py
# results/time.png, speedup.png, efficiency.png + сводная таблица в stdout
```

### 4. Исследование schedule (пункт 3 ЛР)

```bash
./scripts/run_schedule.sh 15000 8 3
# N=15000, threads=8, repeats=3
# Перебирает: static / static,1 / dynamic / dynamic,1 / guided / auto
```

Результаты — в `results/schedule.csv`.

### 5. Привязка к ядрам / NUMA

Скрипт уже использует `OMP_PROC_BIND=close OMP_PLACES=cores`. Если на узле
несколько NUMA-нод, имеет смысл сравнить с привязкой к одной ноде:

```bash
numactl -N 0 -m 0 ./bin/parallel_v1 15000
numactl -N 0 -m 0 ./bin/parallel_v2 15000
```

Для запуска на конкретных ядрах:
```bash
taskset -c 0-7 ./bin/parallel_v1 15000
```

## Замечания по реализации

- **Один и тот же результат на любом числе потоков**: инициализация
  массивов детерминированная (значения зависят только от индексов),
  не зависит от порядка обхода. Сходимость проверяется по `||Ax-b||/||b|| < ε`.
- **Параллельная инициализация** (first-touch) — важно для NUMA: страницы
  попадают в локальную для потока память; без этого V1/V2 на многоядерных
  серверах будут существенно медленнее.
- **V2 — корректность барьеров**: после `single`, который обнуляет
  `y_norm_sq_shared`, есть неявный барьер; редукция собирается через
  `#pragma omp atomic`, после неё — явный `#pragma omp barrier`,
  затем `single` для проверки критерия; после `for` для обновления `x` —
  неявный барьер.

## Замеры на GPU-узле

Задание чисто CPU-OpenMP, GPU не задействован. На узле с NVIDIA GPU
запуск производится на CPU как обычно. Перед замерами убедись, что узел
свободен от других задач (`top`, `htop`), иначе результаты будут шумными.
